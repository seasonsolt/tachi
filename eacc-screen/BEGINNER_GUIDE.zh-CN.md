# eacc-screen 新手使用手册

这份手册只讲当前 `eacc-screen` 这条实际可用的 web 体验，不是假想中的营销官网，也不替你展开所有进阶能力。目标很简单，先让你第一次用的时候不迷路，能尽快看到真实数据，知道主界面在看什么，掉进 `WEB MODE` 以后也知道怎么回到默认的 `cli` 路径。

## 1. 先知道三件事

第一，这个界面是给桌面端用的，宽度小于 `768px` 时不会给你完整体验。  
第二，第一次打开如果还没有数据，直接点 altar 任意区域就会弹出 `Setup`。后面再想改配置，不用等它自己弹，右下角一直有 `Configure` 按钮。  
第三，默认主路是本地 `cli` server，不是 `web`。`WEB MODE` 只是本地 WebSocket 连不上时的 fallback。

## 2. 第一次用，最短路径是什么

如果你只是想先把这个界面跑起来，按这个顺序走就够了：

1. 在桌面浏览器打开页面。
2. 点一下中间 altar，或者直接点右下 `Configure`，打开 `Setup`。
3. 先看 `Local Server` 这一栏。
4. 如果你本地已经有 CLI server，在 `Server Address` 里填 `localhost:3666`，然后点 `Connect`。
5. 页面会 reload。回来以后，如果不再显示 `WEB MODE`，并且 `Local Server` 显示 `Connected`，说明你已经回到了默认 `cli` 路径。
6. 如果你暂时没有本地 server，也可以先走 API usage 路线，在 `Sources` 那块填你要用的 API key，再点 `Save`。
7. 等几秒，看中间 token 数、左下 `Pulse`、右上状态有没有开始动。

先别急着碰 `Market Rite`、Audio、Focus、Theme。它们都不是第一次成功使用的前置条件。

## 3. 如果你看到 `WEB MODE`

这个状态不是“另一个正式入口”，它更像一个兜底模式，意思是页面已经放弃当前的本地 WebSocket 连接，转成浏览器侧还能维持的那套体验。

你会看到两个明显信号：

- `Setup` 顶部出现 `WEB MODE`
- `Claude Code` 会被标成 `CLI mode only`

如果你想回到默认 `cli` 路径，动作很固定：

1. 打开 `Setup`
2. 找到 `Local Server`
3. 填本地地址，比如 `localhost:3666`
4. 点 `Connect`
5. 等页面 reload

回来以后看两个成功信号：

- `WEB MODE` 不再显示
- `Local Server` 显示 `Connected`

只要这两个都出现了，就说明你已经回到 `cli` 路径，接下来 `Sessions` 之类的 CLI 条件内容才有机会出现。

## 4. 主界面怎么读

先别把这个页面当 dashboard 看，它更像一个一直挂着的 ambient screen。真正重要的区块不多。

### Altar

中间最大那串数字就是主 token 计数。  
如果这里只有 `—`，通常不是“坏了”，而是你还没接上数据源，或者还停在没有真实数据的状态。

### Scripture

上方那句轮播文案是氛围层，不是功能提示。它会换，但不影响你判断系统有没有连上。

### Pulse

左下角的 `RATE`、`TODAY`、`MONTH` 是你最该先看的统计区。  
想确认系统是不是真的有数据，先看这块，比看装饰层靠谱。

### Focus

右上角是专注计时器。它跟数据接入没有强依赖，先不用把它当成主线。

### Chant

右下角是环境音控制。默认也不是主线，想开的时候再开就行。

### Sessions

`Sessions` 只有在默认 `cli` 路径成立，而且确实检测到 session 的时候才会出现。  
如果你没看到它，不代表页面坏了，很多时候只是你还没回到 `cli`，或者当前没有可展示的 session。

## 5. `Setup` 面板每一块是干什么的

### Local Server

这是最重要的一块。你想让页面回到默认 `cli` 路径，或者第一次直接连本地 server，都先看这里。

### Sources / API keys

这里决定页面能不能拿到外部 usage 数据。  
在 `web` 模式下，key 会存在浏览器本地；在 `cli` 路径下，配置会发给本地 server。

### Audio

这里是环境音。支持默认 ambient、YouTube 链接和本地音频文件。  
YouTube 走嵌入播放器，本地文件只留在当前浏览器会话里。

### Focus

这里管专注时长、开始、暂停和重置。  
它本身不影响数据接入，放在第一次使用之后再配更省心。

### Theme

这里管主题切换。  
它会先写浏览器本地状态；如果当前已经连上 WS，再往共享链路里同步，所以你有时会觉得“切换已经生效了，但别的端还没跟上”，这不是一个层级的事。

## 6. 几个最常见的新手目标

### 我只想先看到真实数据

优先顺序就是：

1. 打开 `Setup`
2. 先处理 `Local Server`
3. 再处理 `Sources / API keys`
4. 回到主界面看 `Pulse` 和中间 token 数

### 我现在就在 `WEB MODE`

别先去填别的，先做 `Local Server > Connect > reload`。  
先把路径拉回 `cli`，很多解释成本会直接少一半。

### 我只想把页面当氛围屏挂着

先把数据接通，再去开 Audio 和 Focus。  
反过来做也不是不行，只是更容易在一堆设置里把主线弄丢。

### 我没看到 Sessions

先确认两件事：

1. 你是不是还在 `WEB MODE`
2. 当前是不是真的有活跃 session 可被收集到

## 7. 第一次用先不用碰什么

先不用碰条件化能力。  
像 `MarketRite` 这种东西，属于条件能力，不影响你第一次成功使用；真要看，先去 `Setup` 里的进阶区，再看附录。

也先别把 Theme sync、Audio source 细节、Focus preset 当成“必须做完”的步骤。它们都不该排在“先看到真实数据”前面。

## 8. 附录：条件内容和容易误会的地方

### 8.1 `Sessions` 为什么有时没有

它不是常驻保证出现的区块。  
只有在默认 `cli` 路径成立，而且底层真拿到了 session，才会渲染出来。

### 8.2 Theme 为什么看起来已经切了，但别的端还没一起变

因为这条链路分两层：

1. 页面先改自己的本地状态
2. WS 已连上时，才会把 theme change 发到本地 server，再走共享同步

所以你看到“当前页已经变了”并不等于“所有端已经同步完成”。

### 8.3 `MarketRite`

它是条件能力，不是新手主路径。  
你第一次用的时候，只需要知道入口在哪，不需要先学 buyer、seller、operator 这些分支。

如果你真要继续看，至少先知道一件事：它不是一个单一路径。它会跟着当前 `marketState` 变，不同模式下看到的区块不一样。

## 9. 基于这份手册，当前最该收敛的流程

这部分不是拍脑袋列问题，是按“手册写到这里开始不得不额外解释”的地方倒推出来的。

### P0

#### 1. `Setup` 职责混在一起

现在 `Setup` 把连接、密钥、音频、专注、主题都塞在一个面板里。  
对新手来说，最大的问题不是项目多，而是它不会告诉你“先做哪一步才能看到真实数据”。

更直接地说，第一次用的人进来以后，应该先被引到 `Local Server` 和 `Sources`，而不是和 Audio、Focus、Theme 平级放着。

#### 2. `web` fallback 和 `cli` 主路关系太隐蔽

你要写手册时会发现，必须专门拿一整节解释：

- `web` 不是正式并列入口
- 它是怎么掉进去的
- 掉进去以后怎么恢复

这说明当前模式心智本身就不够直。

### P1

#### 3. `Setup` 的双入口不够显眼

第一次是点 altar 触发，之后是右下 `Configure` 复入。  
这两件事在代码里很清楚，但在用户心智里不自然，所以手册必须专门解释。

#### 4. “成功连接”缺少更直接的引导

现在你得自己总结出成功信号：

- `WEB MODE` 消失
- `Local Server` 变成 `Connected`

如果页面自己把这套反馈讲得更直，新手手册会短很多。

#### 5. `MarketRite` 入口和能力层级不够分层

它现在是一个可见入口，但内部其实混着 buyer、seller、operator 这些条件路径。  
所以手册正文里必须刻意把它压下去，不然新手会误以为这也是上手必经步骤。

### P2

#### 6. Audio、Focus、Theme 太早和主线并列

这些都不是没价值，而是出现时机不对。  
第一次使用阶段，它们会放大噪音，拉长“看到真实数据”的时间。

#### 7. Theme sync 的双层语义不直观

切主题这件事看起来像一个动作，实际上分成本地变化和共享同步两层。  
只要手册不得不解释“为什么这边变了，别的端没变”，就说明这条链路对用户来说还不够直。

## 10. 如果下一步真要改，我会怎么排

不用一下子做大重构，按顺序收就够了：

1. 先把 `Setup` 里的主线和非主线分层，至少把 `Local Server`、`Sources` 往前提
2. 再把 `WEB MODE` 的识别和恢复提示做得更直接
3. 然后处理 `MarketRite` 的入口分层，让它更像进阶功能，不像上手主线
4. 最后再收 Audio、Focus、Theme 这些“不是第一次成功的前置项”

说到底，这个页面现在不是不能用，问题是新手第一次进来时，得自己替产品做一轮排序。手册能帮他过这一关，但更好的做法，还是把这轮排序尽量做回产品里。
